Implementing a root Certification Authority with OpenSSL

1. Creating and managing a root Certification Authority with an OCSP responder

Based on the previous works at:

  • https://www.debian-administration.org/article/618/Certificate_Authority_CA_with_OpenSSL
  • http://darizotas.blogspot.com.es/2013/03/understanding-openssl-capl.html
  • https://jamielinux.com/docs/openssl-certificate-authority/index.html
  • https://www.phildev.net/ssl/opensslconf.html

A certificate authority (CA) is an entity that signs digital certificates. If you operate a website, mail server or similar, you should (even MUST) use TLS with a valid certificate for the sake of security. Many servers pay an internationally trusted CA (e.g. VeriSign, DigiCert) to sign a certificate for their domain. Using certificates issued by 'world-wide accepted' CAs makes all major clients (say browsers, OS, mail applications, etc.), which already trust those CAs, to be able to verify our certificate. Since 2016, there is a CA called Let's Encrypt, a non-profit CA providing TLS certificates for free to websites. Since Let's Encrypt is also trusted by all the major web browser and OS, there is no excuse today not to enable TLS on your webserver.

In some cases it may make more sense to create your own CA, rather than using the services of the 'world-wide trusted' ones. Common cases include securing an intranet website, or for issuing certificates to clients to allow them to authenticate to a server (e.g. web or VPN servers).

In the next sections we are going to use OpenSSL in order to create and manage a root CA and an online certificate-status protocol (OCSP) responder where to verify the revocation status of the certificates issued by your CA.

Typically, the root CA does not sign server or client certificates directly. The root CA is only ever used to create one or more intermediate CAs, which are trusted by the root CA to sign certificates on their behalf. This is best practice. It allows the root key to be kept offline and unused as much as possible.

Once the CA is working, we will create a webserver using a certificate issued by our CA and test the certificate and its revocation status with a web browser.

Before proceeding, prepare your virtualized environment:

  1. Create 2 different virtual machines: one for the CA and its OCSP responder and other one for the webserver.
  2. Connect the two VMs to the same NAT network
  3. Change the hostnames to be ca for the CA VM, and webserver for the webserver one.

There are two steps involved when changing the system hostname. First, set the new hostname using the hostnamectl set-hostname command, and then update the /etc/hosts file with the new hostname. For example, to change the system hostname to ca, you would first do:

kali@kali:~$ sudo hostnamectl set-hostname ca
[sudo] password for kali:
kali@kali:~$

and then update your /etc/host file to be (change kali for ca):

127.0.0.1       localhost
127.0.1.1       ca

# The following lines are desirable for IPv6 capable hosts
::1     localhost ip6-localhost ip6-loopback
ff02::1 ip6-allnodes
ff02::2 ip6-allrouters

After changing the hostname, if you open a new terminal you will see the updated prompt.

kali@ca:~$

For the rest of the document we are assuming the following network architecture:

  • CA VM: 10.0.2.15/24, default gw 10.0.2.1
  • Webserver VM: 10.0.2.7/24, default gw 10.0.2.1

Please keep in mind that your actual IP addressing may be different since IP addressing by default in a virtualbox NAT Network is dynamic

1.1. OpenSSL's configuration explained

OpenSSL is an open-source project initially aimed at providing a robust, commercial-grade, and full-featured toolkit for the Transport Layer Security (TLS) and Secure Sockets Layer (SSL) protocols. However, it has become the most widely-used, general-purpose cryptographic library. It is organized in modules called commands facing from basic cryptographic functions (such as symmetric encryption, public-key encryption, digital signature or hash functions) to more complex functionalities (such as managing a CA, managing certificates, creating an OCSP server, etc.).

The OpenSSL configuration file is called openssl.cnf by default and belongs in the /etc/ssl/openssl.cnf path as default in most Linux distributions. However, you can specify a different configuration file by using the OPENSSL_CONF environment variable or you can specify alternative configurations within one configuration file.

The configuration file is an INI-type text file, which means sections begin with [section_name] and run until the next section begins. Anything within a section is a simple key=value pair. There is one additional caveat. Sometimes a key's value is expected to be a section name. This means there is no finite list of possible sections that the parser understands.

Mainly the contents of the openssl.cnf file are related to the following OpenSSL commands:

  • openssl ca, for managing a CA;
  • openssl req, for managing the creation of certificate signing requests (CSR) or simply requests;
  • and openssl tsa, for managing a time-stamp authority (TSA).

We are going to omit the latter (openssl tsa) since it is not used for the purpose of this tutorial.

1.1.1. Configuration of the openssl ca command

The main section with regard to the openssl ca command is (surprise) the ca section:

[ ca ]
default_ca      = CA_default

The ca section defines the way the CA acts when using the openssl ca command, which is in charge of addressing the common CA's functionalities. However, the only thing that is in the CA section is the name of the default CA's section, which in this case is CA_default. We could have several ca sections, each specifying a different configuration for a different CA, and switch between them by changing the default_ca option. We could also override this choice from the command line, using the -name parameter with another section name when invoking openssl ca. This is useful in development and testing, enabling us to try out different configurations.

The CA_default section should look something like:

[ CA_default ]

dir             = ./demoCA              # Where everything is kept
certs           = $dir/certs            # Where the issued certs are kept
crl_dir         = $dir/crl              # Where the issued crl are kept
database        = $dir/index.txt        # database index file.
#unique_subject = no                    # Set to 'no' to allow creation of
                                        # several certs with same subject.
new_certs_dir   = $dir/newcerts         # default place for new certs.

certificate     = $dir/cacert.pem       # The CA certificate
serial          = $dir/serial           # The current serial number
crlnumber       = $dir/crlnumber        # the current crl number
                                        # must be commented out to leave a V1 CRL
crl             = $dir/crl.pem          # The current CRL
private_key     = $dir/private/cakey.pem# The private key

x509_extensions = usr_cert              # The extensions to add to the cert

# Comment out the following two lines for the "traditional"
# (and highly broken) format.
name_opt        = ca_default            # Subject Name options
cert_opt        = ca_default            # Certificate field options

# Extension copying option: use with caution.
# copy_extensions = copy

# Extensions to add to a CRL. Note: Netscape communicator chokes on V2 CRLs
# so this is commented out by default to leave a V1 CRL.
# crlnumber must also be commented out to leave a V1 CRL.
# crl_extensions        = crl_ext

default_days    = 365                   # how long to certify for
default_crl_days= 30                    # how long before next CRL
default_md      = default               # use public key default MD
preserve        = no                    # keep passed DN ordering

# A few difference way of specifying how similar the request should look
# For type CA, the listed attributes must be the same, and the optional
# and supplied fields are just that :-)
policy          = policy_match

From the section, we can clearly see that openssl ca command needs the following directories and files to be able to operate:

  • certs/ - Directory where to store valuable certs.
  • crl/ - Directory where to store the generated CRLs
  • newcerts/ - Directory where OpenSSL stores the certificates issued as PEM files with filename as <#serial>.pem. As our CA is a root CA (with a self-signed certificate), the CA certificate will be the first to appear in this dir. The certificate file name corresponds to its serial number (registered in index.txt).
  • private/ - Directory that stores the CA private key.
    private/cakey.pem - CA private key (PEM format)
  • cacert.pem - CA certificate (PEM format)
  • crlnumber - Stores the number of the current CRL
  • index.txt - Database (text file format) that contains the info of the issued and revoked certificates. It consists of zero or more lines, each containing the following fields separated by tab characters:
    • Certificate status flag (V=valid, R=revoked, E=expired).
    • Certificate expiration date in YYMMDDHHmmSSZ (Z = Zulu time/UTC) format.
    • Certificate revocation date and reason in YYMMDDHHmmSSZ[,reason] format. Empty if not revoked.
    • Certificate serial number in hex.
    • Certificate filename or literal string unknown.
    • Certificate distinguished name (DN).
  • serial - Stores next incremental serial number in hex.

Comments are in general self-explanatory, but let us pay more attention to some other key-values.

x509_extensions = usr_cert              # The extensions to add to the cert

This defines the section in the file to find the x509v3 extensions to are added by default to a certificate when the CA signs a certificate signing request (CSR) or just request. Obviously, we can override this value from the command line passing the option -extensions with a section name to the openssl ca command.

The usr_cert section contains the default set of extensions for a user certificate and will look like this:

[ usr_cert ]
# These extensions are added when 'ca' signs a request.
# This goes against PKIX guidelines but some CAs do it and some software
# requires this to avoid interpreting an end user certificate as a CA.
basicConstraints=CA:FALSE

# This is typical in keyUsage for a client certificate.
# keyUsage = nonRepudiation, digitalSignature, keyEncipherment

# And this for a client certificate that its is goin to be used to authenticate
# in a TLS Server
# extendedKeyUsage=clientAuth

# This will be displayed in Netscape's comment listbox.
nsComment                       = "OpenSSL Generated Certificate"

# PKIX recommendations harmless if included in all certificates.
subjectKeyIdentifier=hash
authorityKeyIdentifier=keyid,issuer

The X509 V3 certificate extension configuration format with all the extensions are detailed in the manual page of x509v3_config, which you can either consult locally with man x509v3_config or online.

Section v3_ca also defines extensions that could be added when signing a request, although now it is targeted to the certificate for a root CA:

[ v3_ca ]
# Extensions for a typical CA

# PKIX recommendation.
subjectKeyIdentifier=hash
authorityKeyIdentifier=keyid:always,issuer
basicConstraints = critical,CA:true

# Key usage: this is typical for a CA certificate. However since it will
# prevent it being used as an test self-signed certificate it is best
# left out by default.
# keyUsage = cRLSign, keyCertSign

Let us get back to the CA_default section and take also a look to the policy key:

policy          = policy_match

The key policy just points to a policy section, in this case policy_match. A policy section specifies to what extent the distinguished name (DN) in a certificate request presented to the CA must agree with the DN of the CA's certificate. It acts as a pre-filtering, so if the request does not meet the policy, it will not be signed. Let's analyse the policy_match section:

[ policy_match ]
countryName             = match
stateOrProvinceName     = match
organizationName        = match
organizationalUnitName  = optional
commonName              = supplied
emailAddress            = optional

With the policy_match policy, our CA will only allow signing certificate requests that supply a commonName (CN) and have the same countryName, stateOrProvinceName and organizationName that are in the DN of the CA's certificate. Obviously, we can switch between different policies by pointing out to another section or using -policy parameter with another policy section name when invoking openssl ca.

By default, there is another policy section called policy_anything where just a CN is required in order to sign a request. Pointing policy to policy_anything will then almost remove any kind of pre-filtering before signing a request. However, this doesn't mean that the request is automatically signed but just that there is no pre-filtering before asking the CA's manager to confirm whether to sign.

[ policy_anything ]
countryName             = optional
stateOrProvinceName     = optional
localityName            = optional
organizationName        = optional
organizationalUnitName  = optional
commonName              = supplied
emailAddress            = optional

1.1.2. Configuration of the openssl req command

Let us analyse another important section in the OpenSSL configuration file: the req section, which configures the openssl req command.

[ req ]
default_bits            = 2048
default_keyfile         = privkey.pem
distinguished_name      = req_distinguished_name
attributes              = req_attributes
x509_extensions = v3_ca # The extensions to add to the self signed cert

# Passwords for private keys if not present they will be prompted for
# input_password = secret
# output_password = secret

string_mask = utf8only

# req_extensions = v3_req # The extensions to add to a certificate request

default_bits and default_keyfile are default values for key length and the private-key's filename respectively.

Note that the x509_extensions refers to another section (v3_ca in this case) that adds x509 extensions to the certificate when the request is self-signed (which is precisely the case of a root CA) instead of being signed by a CA, which would add the extensions defined in the x509_extensions key in the ca section (which usually points to the usr_crt section) instead.

The distinguished_name (DN) key refers to another section (by default req_distinguished_name), which specifies the DN fields required when the openssl req command is creating a certificate request.

[ req_distinguished_name ]
countryName                     = Country Name (2 letter code)
countryName_default             = AU
countryName_min                 = 2
countryName_max                 = 2

stateOrProvinceName             = State or Province Name (full name)
stateOrProvinceName_default     = Some-State

localityName                    = Locality Name (eg, city)

0.organizationName              = Organization Name (eg, company)
0.organizationName_default      = Internet Widgits Pty Ltd

# we can do this but it is not needed normally :-)
#1.organizationName             = Second Organization Name (eg, company)
#1.organizationName_default     = World Wide Web Pty Ltd

organizationalUnitName          = Organizational Unit Name (eg, section)
#organizationalUnitName_default =

commonName                      = Common Name (e.g. server FQDN or YOUR name)
commonName_max                  = 64

emailAddress                    = Email Address
emailAddress_max                = 64

In a DN section, we manage the DN fields that are included in a request and their default values. A default value can be generated by appending _default to the name of the field. For example if we wanted to set default locality_name to Sidney, we would add:

locality_name_default = Sidney

1.2. Create a root CA

We are going to create a root CA in the directory rootca in your home directory. Open a terminal and create that directory and move there.

kali@ca:~$ mkdir rootca
kali@ca:~$ cd rootca
kali@ca:~/rootca$

Instead of modifying the OpenSSL's general configuration file we are going to use a custom configuration for our root CA. Copy /etc/ssl/openssl.cnf to the rootca dir:

kali@ca:~/rootca$ cp /etc/ssl/openssl.cnf .
kali@ca:~/rootca$

1.2.1. Prepare the CA's OpenSSL configuration file

Go to the CA_default section in your version of openssl.cnf. The default configuration almost completely meets our needs. You just need to change tow keys:

  • Change CA's base directory dir to ., so that the openssl ca looks for the CA files and directories in the same directory it is run. As a consequence, our CA will only work if the openssl ca command is run from rootca.
  • Uncomment the line copy_extensions = copy, which allows us to add extensions from a certificate request to the final certificate. As later explained in section 2.1. Getting a certificate for your webserver, this is required in order to support the subject alternative names (SAN) extension.
[ CA_default ]

dir             = .                     # Where everything is kept
...
# Extension copying option: use with caution.
copy_extensions = copy
...

Our root CA will have a self-signed certificate that is created using the openssl req command which is configured though the req section. Let us change the default bit length for the keys generated by that command to 4096, which is the security recommendation for a root CA.

[ req ]
default_bits  = 4096
...

The next few sections define extensions that are going to be applied when signing certificates with our CA.

Section v3_ca already exists in the file and defines extensions for a typical CA. However, it is thought to be used by default for self-signed certificates with any purpose, for example a test webserver certificate. This is nice since self-signed certificates are often only used for testing purposes. However, we just want to create a self-signed certificate for our root CA, so it is a security recommendation to delimit the purpose of the certificate. Change the section to:

[ v3_ca ]
# Extensions for a Root CA (`man x509v3_config`).
basicConstraints = critical,CA:true
subjectKeyIdentifier=hash
authorityKeyIdentifier=keyid:always,issuer
keyUsage = critical, cRLSign, keyCertSign

The usr_cert extension is also in the default configuration but now we want to be make it more specific and restrictive. This extension will be from now on only added to TLS client certificates, which are used when the clients authenticate to a TLS server.

Note that, in order to use OCSP to verify the status the generated client certs, the CA must encode the OCSP server location into the certificates that it signs. In this tutorial we are assuming that the OCSP responder listens on http://10.0.2.15. Obviously, you should update this to meet your configuration.

The usr_cert section should be then modified to:

[ usr_cert ]
# Extensions for client certificates (`man x509v3_config`).
basicConstraints       = CA:FALSE
subjectKeyIdentifier   = hash
authorityKeyIdentifier = keyid, issuer
authorityInfoAccess    = OCSP;URI:http://10.0.2.15
keyUsage               = critical, nonRepudiation, digitalSignature, keyEncipherment, keyAgreement
extendedKeyUsage       = critical, clientAuth

The usr_cert section is going to be applied just for TLS client certificates, so we need to create a new section for the TLS servers as well. Once again, the URI of the OCSP responder will be added to the issued certificates.

[ server_cert ]
# Extensions for server certificates (`man x509v3_config`).
basicConstraints       = CA:FALSE
subjectKeyIdentifier   = hash
authorityKeyIdentifier = keyid, issuer
authorityInfoAccess    = OCSP;URI:http://10.0.2.15
keyUsage               = critical, nonRepudiation, digitalSignature, keyEncipherment, keyAgreement
extendedKeyUsage       = critical, serverAuth

We need also to create extensions for OCSP server certificates, which will be used by the OCSP responders to sign the OCSP responses.

[ ocsp_responder_cert ]
# Extension for OCSP signing certificates (`man ocsp`).
basicConstraints = CA:FALSE
subjectKeyIdentifier = hash
authorityKeyIdentifier = keyid,issuer
keyUsage = critical, digitalSignature
extendedKeyUsage = critical, OCSPSigning

1.2.2. Prepare the required directories and files

As explained before, the openssl ca command expects a directory/file structure as defined in the ca section of the OpenSSL configuration file. In addition to that structure, we will create a directory requests where we are going to store certificate requests to our CA. The final structure will be:

rootca/

 certs/
 crl/
 crlnumber
 index.txt
 newcerts/
 private/
  cakey.pem
 requests/
 serial

Let us first create the required directories and make the private one (where the private key is stored) only accessible to the CA's user.

kali@ca:~/rootca$ mkdir certs crl newcerts private requests
kali@ca:~/rootca$ chmod 700 private
kali@ca:~/rootca$

And now, let us prepare the files:

  • index.txt is initially just an empty file.

    kali@ca:~/rootca$ touch index.txt
kali@ca:~/rootca$

  • crlnumber stores the number of the current CRL, which now is 00.

    kali@ca:~/rootca$ echo 00 > crlnumber
kali@ca:~/rootca$

Three files are still left: the CA's certificate cacert.pem, the private key private/cakey.pem and the serial.

We will first create the CA's private key private/cakey.pem along with its associated certificate signing request requests/careq.pem. This is done with the openssl req command in just one step, but let us first understand what the command will do.

First, it will create the keys. By default, OpenSSL encrypts the private key based on a user-provided password, so that in order to use the private key, it is necessary to provide the password. This is why you will be first asked for a password. Do not forget it, you need it to later operate your CA!

Second, it will generate the certificate request with the Subject's information. That is why you will be asked to fill the DN fields as well as some additional attributes. The Common Name field of the DN is recommended to be and identifier of the purpose. For example, for a domain name, you use the exact domain name; for an email, the email; etc. Since we are creating a root CA, we will use a CN that clearly states it, e.g. NS Root CA.

1.4. Testing our CA + OCSP responder

For testing our CA + OCSP responder, we are going to carry out the following steps:

  1. Issue a test TLS-server certificate.
  2. Verify the certificate against the OCSP responder. It should output that it is a valid (not revoked) certificate.
  3. Revoke the test TLS-server certificate.
  4. Verify the certificate against the OCSP responder. It should output that the certificate is not valid (revoked).

  • TLS client: Any web browser.

2.1. Getting a certificate for your webserver

In your webserver VM you have to generate a certificate signing request to be later sent to the CA one.

We learnt that the Common Name of the certificate should be a fully-qualified domain name, that is to say that if your server operates at https://example.com, the certificate's CN had to be 'example.com'. However, the CN does not support multiple values and thus multiple different certificates had to be created for example for: example.com, mobile.example.com and example.net.

In order to overcome that limitation, the subject alternative name (SAN) extension was proposed. This extension allowed specifying additional subject identities called subject alternative names for the same certificates. Names include:

  • Email addresses
  • IP addresses
  • URIs
  • DNS names: this is usually also provided as the Common Name RDN within the Subject field of the main certificate.
  • Directory names: alternative Distinguished Names to that given in the Subject.
  • Other names, given as a General Name or Universal Principal Name: a registered object identifier followed by a value.

Soon, SAN was supported by all major clients (i.e. web browsers), and nowadays, based on the recommendations of RFC 3200, every certificate must use the subjectAltName extension. In fact, the CN is only evaluated, if evaluated, if subjectAltName is not present, and only for compatibility with old, non-compliant software.

Let us prepare the server to add the alternative names to our certificate request. Since we don't own a domain name for this tutorial, we are going to use the webserver's IP, namely 10.0.2.7.

'use strict';

const httpsOptions = {
    key: tlsServerKey,

const server = https.createServer(httpsOptions, app);

/**

server.listen(443);
server.on('listening', onListening);

/**
 * Event listener for HTTP server "listening" event.
 */
function onListening() {
    const addr = server.address();
    const bind = typeof addr === 'string'
        ? 'pipe ' + addr
        : 'port ' + addr.port;
    console.log('Listening on ' + bind);
}

3. Do-it-yourself tasks

3.1. Test your TLS webserver with a web browser

Since your web server is only accessible from the virtualbox natnetwork it is connected to, you need to run a browser in a virtual machine connected to that same network. You could use a new VM or just open a web browser in either the CA or the webserver VM. Then, follow the next steps:

  • Install the CAs certificate on the clients web browser.
  • Connect the browser to the server https://10.0.2.7.
  • Check that the connection does not raise any warnings (unless the servers certificate had an inappropriate subjectAltName extension).
  • Check also that the web browser has also queried your OCSP responder to check the status of the server's certificate.
  • Revoke the webserver's certificate and check that the browser is not letting you in since the certificate is revoked.

3.2. Authenticate to your webserver using a client certificate

3.2.1. Prepare your webserver

A couple of hints:

  • You need to pass a new option to the Node's https module so that it requires client certificates.
  • You need also to tell the https server where the certificate of your CA is so that it can properly verify the clients' certificates.

3.2.2. Prepare your client

  • Generate a certificate for the client (it should have the usr_cert extensions).
  • Generate a PKCS #12 file (typical extension is .p12 or .pfx) containing the client certificate and private key. Clue: Use the openssl pkcs12 command.
  • Import the PKCS 12 file to your browser so it can use it as a TLS client certificate.
  • Connect to your webserver and check that you can use your certificate to authenticate.
